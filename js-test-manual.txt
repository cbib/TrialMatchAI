JavaScript String conversion test:
String(null) = "null"  // string literal containing 4 characters
String(undefined) = "undefined"  // string literal containing 9 characters

So if pt.id = null:
String(pt.id) = "null" = "null"
href = "#/p/null"

In route():
location.hash = "#/p/null"
const m = location.hash.match(/^#\/p\/(.+)$/)
// m[1] = "null"
const id = decodeURIComponent(m[1])
// id = "null"

Then:
const p = PATIENTS.find(x => String((x.patient || {}).id) === id)
// Looking for: String(x.patient.id) === "null"
// If x.patient.id = null, then String(null) = "null"
// So: "null" === "null" → TRUE
// MATCH FOUND

But if another patient has id = "null" (string):
// String("null") = "null"
// So: "null" === "null" → TRUE
// ALSO MATCHES (collision!)

This is the real bug: 
- If you have a patient with id=null and another with id="null", routing breaks
- String representation creates ambiguity
- The display shows "Patient —" for null, but routing uses "null" href

The fix should be:
- Don't route to overview rows with null id
- Or use a safer encoding that distinguishes null from "null"

Actually, looking at line 330 again:
pt.id could be:
- null (no patient id)
- undefined (missing patient object)
- string value like "patient123"
- number like 123

The issue: String(null) and String(undefined) both create string values that could collide with real patient IDs.

Better fix: Check for null/undefined before creating href:
```javascript
const row = el("a", { 
  class: "prow", 
  href: pt.id != null ? "#/p/" + encodeURIComponent(String(pt.id)) : "#/"
});
```

This way, patients with null/undefined id just go to the overview (#/) instead of creating a broken "#/p/null" href.
